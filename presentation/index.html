Zero-Knowledge Proofs

A party (prover) convinces another party (verifier) that a statement is true without revealing any information aside from the truth of the statement.

Completeness

Honest prover can always convince verifier

Soundness

Dishonest prover cannot cheat

Zero-Knowledge

Verifier learns nothing beyond validity

On blockchain: guarantees state changes with anonymity — e.g., ZCash transfers tokens without disclosing sender, recipient, or amount

Preliminary

Merkle Trees & Nullifiers

Verify membership with O(log n) proof

Nullifiers prevent double-spending

Prove: H(n || s) ∈ MerkleTree   &&   NullifierHash = H(n)   // without revealing position

Case Study

Tornado Cash

Non-custodial mixer: deposit and withdraw Ether with no reference to the deposit address. Pioneer of shielded pools with zk-SNARKs (Pertsev et al., 2019).

How it works

Cryptocurrency enters a shielded pool and can no longer be tracked until after exiting. Uses fixed denominations (0.1, 1, 10, 100 ETH).

Limitation

FIFO correlation heuristics can undermine anonymity. Tang et al. (2021) linked deposits to withdrawals via timing patterns.

Key insight: Shielded pool model works for privacy — but Tornado Cash is only a mixer, not a trading venue

Design

Design Choices

TEE for Matching, ZK for Withdrawal

Sub-millisecond matching vs ~1 min ZK proof. Fund safety must not depend on TEE integrity.

Shielded Pool over Account Model

No persistent pseudonymous identity. Fresh commitments break linkability.

SP1 over Circom / Halo2

Same Rust code compiles for native + in-circuit. Eliminates commitment-scheme divergence.

keccak256 over Poseidon

Native EVM opcode (30 gas). Deposit 3.8× cheaper than Tornado Cash's MiMC.

Challenges

Implementation Challenges

Cross-Language Commitment Compatibility

Solidity abi.encodePacked vs Rust byte concatenation. Required strict fixed-padding spec + cross-language test vectors.

Merkle Tree 3-Way Consistency

Solidity, Rust, and SP1 must agree on depth-20, zero values, sibling ordering. Built FFI test suite (vm.ffi()).

Root Staleness Under Concurrent Deposits

Proof takes ~1 min. If >30 deposits in that window, root expires. 30-root ring buffer tradeoff.

SP1 no_std Constraints

RISC-V with no standard library. keccak256 + Merkle tree as pure no_std Rust, zero heap allocation.
